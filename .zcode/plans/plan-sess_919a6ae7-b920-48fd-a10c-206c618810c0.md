## Fix Black Blank Screen in Release APK

### Root Cause Analysis
The release APK shows a black blank screen because of **3 compounding issues**:

1. **Over-aggressive R8 optimization** strips/renames classes that GeckoView and Compose need via reflection (`-repackageclasses ''` + `-allowaccessmodification` + 5 optimization passes)
2. **Crash-loop masking**: The uncaught exception handler (line 127) catches crashes, restarts the app, but the user only sees the `#070A0F` (near-black) `windowBackground` — no crash dialog, just a black screen
3. **Unsafe synchronous Gecko init**: `getGeckoRuntime()` is called directly inside `setContent {}` composition (line 175), so any crash there kills the entire UI tree silently

### Changes

#### 1. Add Compose + Gecko keep rules to `app/proguard-rules.pro`
- Keep all `androidx.compose` classes (R8 can strip Compose internal classes that are loaded via reflection)
- Keep `GeckoResult` and its inner classes (used heavily by GeckoView via reflection)
- Keep all WebExtension delegate callback class members
- Keep `org.json` classes (used for extension message passing)
- Keep `kotlin.coroutines` classes that GeckoView's async APIs depend on

#### 2. Tone down R8 optimization in `app/proguard-rules.pro`
- Reduce optimization passes from 5 → 3
- Remove `-allowaccessmodification` (allows R8 to widen access, breaking reflection-based GeckoView internals)
- Remove `-repackageclasses ''` (moves classes to root package, breaks GeckoView's classpath assumptions)
- Narrow the `-assumenosideeffects` for `kotlin.jvm.internal.Intrinsics` — removing null checks in release can cause NPE crashes that the crash-loop handler masks

#### 3. Fix crash-loop handler in `MainActivity.kt`
- Add a crash counter in SharedPreferences — if the app crashes more than 2 times within 60 seconds, stop auto-restarting and show a "crashed" Toast/dialog before exiting cleanly
- This prevents the infinite black-screen restart loop

#### 4. Wrap `getGeckoRuntime()` call in try-catch in `MainActivity.kt`
- The call at line 175 (`browserViewModel.getGeckoRuntime(this)`) runs inside Compose composition with no error handling
- Wrap in try-catch so a Gecko init failure doesn't crash the entire composition — instead show a user-friendly error screen
